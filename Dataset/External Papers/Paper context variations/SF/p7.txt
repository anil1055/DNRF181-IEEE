**Abstract** Analysing, scheduling, developing, testing, and maintaining are the main
activities of software project management (SPM) under software engineering. Software size is the major factor determining cost, time, and effort for developing software
applications. The successful completion of all the activities of SPM requires an accurate determination of the cost, time, and effort required for developing the application.
That is, the real size of the software application is the major determinant of all the
SPM activities. The existing traditional application-oriented and single-dimensional
software sizing techniques will not give the actual size of modern software applications like artificial intelligence (AI) and big data analytics (BDA). The imperfect
estimation leads to wrong predictions about the cost, time, and effort required for
software application development. The wrong estimates will affect the entire SPM
activities, resulting in a delay in delivery, a loss in predicted cost, an imbalanced
effort level, and the dismissal of the software development process. The major software industries are facing the same problem with wrong size estimates. The updated
and dynamic new size estimation technique is highly essential for this current SPM
environment. The novel approach of modern metrics (MM) addresses all the technical challenges facing software engineering for the software size estimation process.
This article analyses the major issues of the software size estimation process, major
functional and non-functional units associated with software size estimation, the
architecture of MM, and an algorithm for software size estimation using MM.

**1** **Introduction**

Software engineering is an engineering procedure for creating software application projects in an international standard manner, giving guidelines for completing
the project on time, testing and debugging in user as well as developer-centric
approaches, and providing the complete procedure for maintaining the developed
software. Modern software systems use about one programming language, application software, utility tool, architecture, platform (operating system), type of user,
and databases. So, the development of a modern software system is highly complex.
Modern applications are founded on software as well as hardware. The systems
(both hardware and software) are generating a huge amount of data. That data must
be processed in parallel with the next process [1].

The modern industry is developing applications in artificial intelligence (AI) and
the Internet of things (IoT). The applications are processing a huge amount of data
(GB to PB) to generate a simple result, which uses different types of datasets (structured, unstructured, and semi-structured), different sets of database management
systems (DBMS), software architectural models, etc. Emerging automation technologies like IoT use sensors, actuators, clouds for storage, big data analytics (BDA)
tools, and highly efficient programming languages. AI techniques such as machine
learning (ML) and deep learning (DL) analyse historical data. To develop such a
complex application requires about one programming language, utility apps, architectural models, DBMS tools (for handling structured, semi-structured, and unstructured data), domain experts (for handling problems in different domains of software),
storage (static and dynamic), parallel processing processors, network topologies,
distributed servers, virtual processing, etc.

Developing and delivering this complex software product on schedule is a very
difficult task. A software engineering study says that less than 30% of the software
is delivered on time. The latency in delivery leads to economic losses for both the
customer and the developer. The trust of the developer is challenged. One of the
important reasons for this delayed delivery is a lack of planning. In the planning
process, software size estimation plays a crucial role. The estimation of effort, cost,
and time for developing the software is determined by the software size factor [2, 3].

Software is an intangible, logical product. So, the estimation of size is highly
risky. The software industry is using many traditional size estimation techniques like
lines of code (LoC), function point (FP), feature point, Delphi, expert judgement,
pattern matching, etc. In terms of software sizing techniques, the FP method is the
most reliable, programming language-independent, user-centric, and understandable
to beginners. The FP method deals with the size of the software at the analysis front
itself. So, the estimation of effort, cost, and time is possible at the initial phase itself.

**3** **Modern Metrics**

Modern metrics (MM) is a proposed novel methodology for identifying the size of
software products like AI and BDA applications. It follows Alan Albrecht’s FPA
method with new and updated function points and complexity adjustment factors
(CAF). This method is language-, platform-, and technology-independent. The resultant value of this method is an important factor for estimating the time, effort, and
total cost of the software product. The estimation process is fully documentable and
repeatable. This method is capable of identifying the size of the software at any
stage of the SDLC. The estimation process gives effective outcomes that are easily
understandable by any kind of user.

***MM Architecture***

The MM is comprised of FPs and CAFs. The MM architecture is illustrated in Fig. 2

II: Internal Inputs, IO: Internal Operations, ILF: Internal Logical Files, EIF:
External Interface Files, CAF: Complexity Adjustment Factors, SDLC: Software
Development Life Cycle, MMSize: Modern Metrics Size, UMMFP: Unadjusted
MM Function Points, and MMCAF: MM Complexity Adjustment Factor.

The architecture explains all of a software’s internal, external, and fusion functional units; both internal and external complexity factors influence the SDLC. The
size of the software application is the multiplied outcome of UMMFP and MMCAF.

F. *Advantages of MM*
The highlighted advantages of MM are as follows:

1. The MM technique is a novel technique for finding the size of the software
in any phase of the SDLC and also finding the size after the completion of
the software development.
2. The MM technique finds the effort required for the development of the functional units in an individual and collective manner. So, the defects present
in the FP count are negligible.
3. The complexity adjustment factors and effort metrics are modified and
updated based on the requirements of the modern software system.

G. *Future Developments*
The future enhancements in MM are as follows:

1. The MM technique was developed based on Indian software industry values.
So, the analysis must be done using international values.
2. The modern software industry is using a greater number of functional units
with a function. So, it is facing difficulties in fixing the limits of the very
high metric of functional values.

**4** **Conclusions**

The modern software industry is waiting for an effective size estimator with updated
parameters for solving software management-based technical, economic, and social
issues. The proposed MM analyses all the issues associated with the software from
a user and developer perspective. It gives the feature of estimating the size of the
software from any of its SDLC phases. The earlier estimation helps the software
industry to do the development process with the derived time, cost, and effort. It
increases the efficiency of delivery. The analysis-based work of the MM is progressive
with industrial parameters.
