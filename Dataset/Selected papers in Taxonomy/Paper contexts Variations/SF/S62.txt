**Abstract**

*Some size measures have been proposed in the literature*
*to be employed in estimating Web application development*
*effort, but to date few empirical studies have been under-*
*taken to validate them and compare their effectiveness. To*
*address this issue we have performed an empirical study by*
*using as size measures COSMIC and Web Objects, with the*
*aim of verifying their effectiveness as indicators of develop-*
*ment effort. In particular, we have built two effort estima-*
*tion models by applying the Ordinary Least-Squares regres-*
*sion and by using a dataset of 15 Web applications devel-*
*oped by an italian software company. The performance of*
*the obtained models have been evaluated by using a dataset*
*of further 4 Web applications developed by the same soft-*
*ware company some time after the first 15 Web applications.*
*The results reveal that both Web Objects and COSMIC are*
*good indicators of the development effort.*

**Index terms** : Empirical Studies, Effort estimation, Web applications, Web engineering.

**1** **Introduction**

Estimating the effort required to develop Web applications represents an emerging issue in the field of Web engineering since it can deeply affect the competitiveness of
a software company [5] [26] [27] [30] [25] [34] [36] [37]

[38]. In the context of traditional software engineering,
several widely accepted (model-based) methods to estimate
software development effort can be found. These methods rely on a formal approach, involving the use of algorithms that take in input some project factors influencing
the development effort (such as the software size) and produce an effort estimate [8]. However, to date, this is not
the case for Web applications, whose development effort is
usually estimated using non-model based methods, mostly
relying on expert judgments. A limitation to the diffusion


of model-based methods comes from the lack of widelyaccepted ways to estimate Web application size that represents a key cost factor. Indeed, the size measures conceived
and widely accepted for traditional software systems, such
as *Function Points* ( *FPs* ) [3], are not suited for Web-based
ones, since they fail to capture some specific features affecting the effort required for applications [36] [37] [38]. In
the last years, some methods have been proposed to address
this issue. Among them, *Web Objects* [34] and *COSMIC*

[14] are of special interest, since in some preliminary case
studies they showed interesting results in predicting Web
application development effort [15] [37] [38] and revealed
their superior performance over standard *FPs* [38]. However it is widely acknowledged that a software measure can
be widely adopted in the industrial domain only if it has
been validated through several studies that have proved its
usefulness [6] [7] [40]. To date, there is a strong need for
further empirical evidence to support these size measures.
In this paper we report on an empirical study aimed at assessing the effectiveness of both *Web Objects* and *COSMIC*
as indicators of Web application development effort. As for
the estimation technique, we exploited a commonly adopted
technique, namely the *Ordinary Least-Squares* ( *OLS* ) *re-*
*gression*, that is a model-based technique [8].
In order to validate the effort estimation we have performed a validation based on an *hold-out sample of applica-*
*tions* . This is theoretically the best option since the validation set is a totally separate dataset from the dataset used to
build the effort estimation models [22]. In particular, in our
study the effort estimation models, one for each size measures, were constructed using 15 Web applications (training set) developed by an Italian software company. Each
model was then validated exploiting further 4 Web applications (test set) developed some time after the first 15. In
other words, the estimated effort for the applications in the
test set are determined using the effort estimation models
obtained using the applications in the training set.

**2** **Employed size measures, estimation tech-**
**nique, and validation criteria**

**2.1** **Web Objects**

*Web Objects* were proposed by Reifer to measure the size
of Web applications [34]. In particular, Reifer adds four
new Web-related components to the five function types of
the *FPA* ( *Function Point Analysis* ) method, namely *Multi-*
*media Files*, *Web Building Blocks*, *Scripts*, and *Links* . In
Table 1 we recall the definition of the Web Objects components as provided by Reifer in his white paper explaining the
counting conventions of the *Web Objects* [35]. He devised
such list based on the opinion of experts and by analyzing
64 Web applications from various application domains [35].
To apply the *Web Objects* method, firstly the instances
of the nine components are counted, and a complexity (low,
average, or high) is associated to them. Then, a weight is
associated to each counted instance of the components, by
using a calculation worksheet [35]. The total amount of
*Web Objects* is obtained according to the weights that are
related to the complexity of each component, giving an indication of the size of the considered Web application. For


further details see [35]. No empirical study was performed
by Reifer to show the effectiveness of this approach. Successively, *Web Objects* have been employed in two case
studies achieving interesting results [37] [38].

**2.2** **COSMIC**

*COSMIC* is a widely adopted method for sizing software,
which has been approved as an international standard by
ISO (ISO/IEC 19761:2003). This method was originally
conceived for business and real-time applications, but it can
be successfully applied in any other context, where software
is mainly dominated by large amounts of data movements.
Conversely, the method is not suited to measure software
mostly characterized by complex algorithms or other specialized and complex rules [13]. The basic idea underlying
this approach is that, for many kinds of software, the most of
development efforts are devoted to handle data movements
from/to the persistent storage and the users. Thus, their
number can provide a meaningful sight of the system size
(see the document: Guideline for sizing business application Software Using *COSMIC-FFP*, version 1.0, December
2005 [13]). To identify data movements, *COSMIC* requires
to define a proper context model for the specific application,
bounding the software from its operating environment. Basically each data movement crossing a boundary has to be
counted. The method considers four kinds of data move
ments, i.e., *Entry*, *Exit*, *Read*, and *Write*, and the functional
size of the software, in terms of *COSMIC Function Point*

[13], is given by the sum of all these data movements.
The application of the *COSMIC* method to size Web systems has been analyzed by some researchers in the last years

[15] [24] [36]. The difficulties of applying the *FPA* method
for sizing an Internet bank system firstly motivated Rollo
to use the *COSMIC* method [36]. The successful application of the method to that system made Rollo confident
that the *COSMIC* approach was suited for sizing Web applications [36]. However, he did not present any empirical
result supporting his thesis. Successively, two studies investigated *COSMIC* approach to estimate Web application
development effort [15] [24], obtaining interesting results.

**2.3** **OLS regression**

*OLS regression* [32] has been widely and successfully
employed in the industrial context and in several researches
to estimate software development effort [8]. This holds also
for Web applications [5] [15] [27] [30] [38].
*OLS regression* is a statistical technique that explores the
relationship between a dependent variable and one or more
independent variables [32], providing a prediction model
described by an equation

(1) *y* = *b* 1 *x* 1 + *b* 2 *x* 2 + *...* + *b* *n* *x* *n* + *c*

where *y* is the dependent variable (the effort), *x* 1 *, x* 2 *, ..., x* *n*
are the independent variables (the cost drivers), *b* *i* is the coefficient that represents the amount the variable *y* changes
when the variables *x* *i* changes 1 unit, and *c* is the intercept.
In our empirical study we have exploited the *OLS regression*
to obtain linear regression models that use the variable representing the effort (in terms of person/hours) as dependent
(namely *EFH* ) and the variable denoting the employed size
measure (namely *WO*, representing *Web Objects*, or *CFP*,
representing *COSMIC* ) as independent. Once the prediction
model has been constructed, the effort estimation for a new
system has been obtained by sizing it in terms of *WO* ( *CFP*,
respectively) and using this value in the equation *EFH* =
*b* - *WO* + *c* ( *EFH* = *b* - *CFP* + *c*, respectively).
Moreover, in order to validate the obtained effort estimation models we have exploited a *hold-out validation* approach, thus a validation based on the use of an *hold-out*
*sample of applications* [22]. In particular, we employed two
different datasets: the first obtained from Web applications
developed by a software company in the past and the second dataset obtained from Web applications developed by
the same software company after the ones included in the
first set. The first dataset (training set) was used to build the
effort estimation models while the second dataset (test set)
was used validate the models.

**2.4** **Evaluation criteria**

The accuracy of the estimations has been assessed using *MMRE* and *Pred* (0.25) [12], and boxplots of absolute
residuals [28]. In the following we briefly recall the main
concepts underlying *MMRE* and *Pred* (0.25), and boxplots.
The *Magnitude of Relative Error* [12] is defined as

(2) *MRE* = *|EFHreal −* *EFHpred|/EFHreal*

where *EFHreal* and *EFHpred* are the actual and the estimated effort, respectively. *MRE* has to be calculated for
each observation in the dataset. To have a cumulative mea
sure of the error, all the *MRE* values have been aggregated
across all the observations, thus obtaining two measures of
the central tendency, known as *Mean of MRE* ( *MMRE* ), and
*Median MRE* ( *MdMRE* ), where the latter is less sensitive to
extreme values [27].
The *Prediction at level* 0.25 [12] is defined as

(3) *Pred* (0 *.* 25) = *k/N*

where *k* is the number of observations whose *MRE* is less

than or equal to 0.25, and *N* is the total number of observations. According to Conte *et al.*, a good effort prediction
model should have a *MMRE≤* 0.25 and *Pred* (0.25) *≤* 0.75,
meaning that at least 75% of the predicted values should
fall within 25% of their actual values [12].


Boxplots are widely employed in data analysis since
they provide a quick visual representation to summarize the
data, using five values: median, upper and lower quartiles,
minimum and maximum values, and outliers [23]. They
represent a usefull graphical representation to compare
the distributions of absolute residuals (i.e., *|EFHreal −*
*EFHpred|* ) obtained with different estimation models.
Moreover, we have tested the *statistical significance* of
the obtained results by using absolute residuals, in order to
establish if one of the measures provided better results than
the others [23] [27]. In particular, we have performed statistical tests to verify whether the rwo population of abosolute
residuals, obtained using *CFP* and *WO*, respectively, have
identical distributions.

**3.3** **Threats to Validity**

It is widely recognized that several factors can bias the
*construct*, *internal*, *external*, and *conclusion* validity of empirical studies. To satisfy construct validity a study has “to


establish correct operational measures for the concepts being studied” [21]. In other words, it represents to what extent the predictor and response variables precisely measure
the concepts they claim to measure [27]. A study has internal validity if its outcome can be considered as a function of the measured variables and not of extraneous fac
tors. Conclusion validity is concerned with the ability to
draw statistically correct conclusions while external validity is concerned with the ability to generalize the results to
other contexts. In the following, we analyze our empirical
study with respect to each of these type of validity.

As for the *construct validity*, the choice of the size measures and how to collect them represents the crucial aspects.
Regarding the selection of the size measures, we employed
two measures that have been used in the past to estimate
Web applications development effort [15] [38]. The collection of information about the application measures and
actual effort represents the main difficulties to carry out this
kind of study. Concerning the effort collection, the software
company used timesheets to keep track of this information,
where each team member annotates the information about

his/her development effort, and weekly each project manager stored the sum of the efforts for the team. In order to
collect all the significant information to calculate the values of the size measures, the authors defined a template to
be filled in by the project managers. All the project managers were trained on the use of the questionnaires. They
took into account the *COSMIC* Measurement manual, version 2.2 [13] and the rules provided in [15] for the collection
required to apply *COSMIC* method. Moreover, they employed the counting conventions of *FPA* method [20] and
followed the suggestions provided by Reifer in his “Web
Objects White Paper” [35] where he explains how to quantify the *Web Objects* components for a given Web application. The projects managers had experience in applying the
*FPA* method so they did not manifest any kind of difficulties
to collect the information required for our empirical analysis. One of the authors analyzed the filled templates and
the analysis and design documents, in order to cross-check
the provided information. The same author calculated the
values of the size measures.

Some factors should be taken into account for the *inter-*

*nal validity* : subjects’ authoring and designing experience,
and reliability of the data and lack of standardization [5]

[6] [23] [27]. The subjects involved in the study were professionals who worked in the software company. No initial selection of the subjects was carried out, so no bias has
been apparently introduced. Moreover, the Web applications were developed with technologies and methods that
subjects had experienced. Consequently, confounding effects from the employed methods and tools can be excluded.
As for the reliability of the data and lack of standardization,
the questionnaires used were the same for all the applica

As for the *conclusion validity* we have carefully applied
the statistical tests, verifying all the required assumptions.
A threat to conclusion validity could be the few number of
Web applications composing the dataset. However, it is recognized that getting information on a great number of real

**4.1** **Empirical studies with COSMIC and**
**Web Objects**

*COSMIC* was applied in two studies that exploited sets
of Web applications developed by students [15] [24], while
*Web Objects* was assessed in two studies [37] [38] exploiting an industrial dataset.
Regarding the studies carried out by Ruhe *et al.*, they
performed an *OLS* regression analysis based on a dataset of
12 Web applications developed by a medium-size software
company operating in Australia and obtained interesting results in terms of prediction accuracy of the method [38].
Moreover, they also used *Web Objects* as size metrics in the
context of *WebCOBRA* [37]. In their studies, they obtained
*MMRE* =0.24, *MdMRE* =0.23, and *Pred* (0.25)=0.67 for the
*OLS* regression model and *MMRE* =0.17, *MdMRE* =0.15,
and *Pred* (0.25)=0.75 for the *WebCOBRA* model. Thus, the
analysis reported in the present paper has confirmed the
goodness of *Web Objects* to estimate effort. Moreover, it
is worth noting that our results not only confirm the results


of Ruhe *et al.*, but also extend those results. Indeed, the applications used in our empirical study have larger sizes than
the applications used by Ruhe *et al.* . In particular, the mean
effort of our datasets is about three times the mean effort

of the Ruhe *et al.* ’s dataset and the applications used in our
empirical study are characterized by a mean number of *Web*
*Objects* about five times the one of the applications used by
Ruhe *et al.* . Thus, the *Web Objects* metrics can be considered a scalable measure in the sense that it turns out to be

good for applications of larger size, too.
Regarding the empirical study with *COSMIC*, Mendes *et*
*al.* constructed an effort prediction model by applying *OLS*
regression analysis and using data on 37 Web hypermedia
applications developed by postgraduate and MSc students
attending a Hypermedia and Multimedia Systems course at
the University of Auckland (NZ) [24]. The derived model
did not present reasonable prediction accuracy and replications of the empirical study were highly recommended
by the authors. In [15] *COSMIC* method has been applied on 44 Web applications developed by academic students and the model obtained with *OLS* regression analysis was characterized by good estimations if we assume as
a reasonable threshold Conte *et al.* ’s suggestion [12]. In
particular, they obtained *MMRE* =0.15, *MdMRE* =0.10, and
*Pred* (0.25)=0.75. However, the scientific literature has often debated on the industrial relevance of results coming
from empirical studies with students [5] [11] [27]. Even if
in the context of Web application development it is recognized that professional developers are usually young programmers, coming straight from university [27], further
analysis exploiting an industrial dataset is needed to verify/confirm the results obtained with the student Web applications. To this end, the empirical studies we present in this
paper was carried out using an industrial dataset. The results
confirmed those obtained in [15], suggesting that *COSMIC*
can be effectively employed to predict Web application development effort.

**5** **Conclusions and lesson learned**

We have reported on an empirical study carried out to
assess the effectiveness of *Web Objects* and *COSMIC* as indicators of Web application development effort and to compare their prediction accuracy. As for the datasets used in
the empirical analysis, we have exploited a set of 15 Web
applications developed by a software company to build two
effort estimation models, while further 4 Web applications
provided by the same software company have been employed to validate the obtained models. In other words,
we have performed a validation based on *an independent*
*hold-out sample of applications* [22]. The results have
highlighted the usefulness of *Web Objects* and *COSMIC*
measures in estimating Web application development effort when employed in combination with *OLS regression* .
The analysis has also revealed that the estimation results
achieved with these measures are comparable.
Finally, observe that the results presented in this paper
have provided some indications for the software company
that supplied the data. They might be also relevant for
other companies that develop Web applications similar to
those used in our investigation. However, a replication of
the study, in different setting and with a larger dataset, is
required in order to generalize the results to other companies. Indeed, it is widely recognized that several investigations should be performed to verify/confirm empirical results [40]. Thus, as future work, we plan to collect and
analyze further data also from other companies. Moreover,
we will investigate other estimation techniques (e.g., *Case-*
*Based Reasoning* [39] that have provided interesting results
for traditional software systems. As suggested in [19] we
also plan to experiment regression models using on one or
more data movement types (i.e., *Exit*, *Entry*, *Write*, and
*Read* ) as independent variables instead of the solely *CFP*
value.
