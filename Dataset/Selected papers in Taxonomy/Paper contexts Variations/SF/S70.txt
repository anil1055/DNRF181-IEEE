***Abstract*** **—This paper compares the effectiveness of Function**
**Points and Web Object analysis to estimate the development**
**effort of Web applications. The empirical study is performed on**
**a set of industrial projects carried on in an Italian software**
**company. Starting from original requirements and estimates,**
**carried on using FP analysis, we compute the Web Objects**
**analysis of the same requirements, and compare the results of the**
**two approaches. We also assess the relative effectiveness of FP**
**and WO using post-mortem data about the actual effort taken**
**to develop the studied systems.**

I. I NTRODUCTION

Estimating software development costs is a central issue
in software engineering; avoiding big mistakes in evaluating
the effort required for developing a system, while satisfying
all the customers’ needs and requested features, is a key
success factor for companies willing to be competitive in the
market. Software development is a complex process involving
a plurality of aspects (team skills, technologies, domain,
stakeholders...), so the estimating activity is affected by many
variables. However, it is widely recognized that size is the
main cost driver in an estimation model [1] [2] so, adopting
the best suited size metric for the systems whose development
effort has to be predicted, is a prerequisite for achieving good
results in this process.
Web technologies, once exploited only for creating hypermedia
applications (static web sites), have known a huge development, and now they are one of the cutting-edge technologies
for very different kinds of systems, from small informationcentred applications to large enterprise-scale commercial systems. The differences from traditional software, in terms of
technology, development model, time-to-market needs, volatility of requirements and so on, pose serious challenges in
adapting traditional size metrics, like Function Points [3], to
measure Web applications.
The purpose of this paper is to compare two size measures,
a traditional one (Function Points), and a recently proposed,
web-aimed one (Web Objects [4]), to evaluate their ability to
predict the effort spent in Web applications development, in
the context of an Italian software company. The main issue


we investigate is whether a specific web-addressed metric
can give a more accurate prediction of size, and therefore
effort, than a consolidated, widespread and general-purpose
functional measure like Function Points.

The paper is organized as follows: Section 2 discusses the
related works in the area of size measurement, while Section
3 describes our study and the followed methodology, and
presents the results of the research. Section 4 discusses these
results, and eventually Section 5 presents the conclusions and
plans for future work.

II. R ELATED WORK

Sizing software systems can be quite a complex task, if
applications developed with new or emerging technologies
must be measured using metrics tailored for different technologies/architectures. This is the case of adapting functional sizing
for Object Oriented systems, where OOFP (Object Oriented
Function Points [5]), Object Points [6], Use Case Points [7]
and Class Points [8] constitutes the most important attempts
in defining a measure where an object decomposition of the
system is much more natural than a functional one.
On the other hand, few examples can be found in literature
about specific size metrics for web applications: Web Points

[9], Internet Points [10], Data Web Points [11] try to address
the problem of sizing web systems from different point of
views. Unfortunately, we found them quite unfit for providing
a reliable metric able to size in an easy and consistent way a
complex web application.
Reifer’s Web Objects [4] is a quite different measure, since it
starts from the most classic functional size metric, Function
Points, and extends it by taking into account components that
can be typically found in a web system. One advantage of this
metric is constituted by the fact that it has an associated cost
model, WebMO. Although its validity has not been empirical
demonstrated yet, it seems the only model specifically tailored
for estimating web applications development effort, as there
are no other models that takes as input a web-specific size
measure (Web Objects in this case).
As a matter of fact, size measures’ main goal is to describe,

project, how large a software system will be; this prediction

such as COCOMO [1] and its evolution COCOMO II [12] use

due to the fact that they have been developed to support

the limitations of these models, several studies evaluated the
effectiveness of models based on analogy, like Case-Based
Reasoning [13] [14] [15], and other statistical approaches [16]

[17], that aim to build a model that relies on the knowledge
of previous projects, whose actual effort is obviously known.
This is the case of the only study that assess the validity
of a model based on Web Objects size estimation [18]; it
examines a number of industrial software projects from the
same company, building a model based on the knowledge of
the actual effort spent on them.
In the next section, the description of the research will try
to underline the different approach we followed to assess the
reliability of Web Objects size measure.

III. R ESEARCH STUDY

The empirical research has been conduced in the context
of a mid-sized Italian software company. Choosing a so
narrow sample for our study (projects developed by only
one company) could constitute a possible threat to generality.
However, the lack of empirical studies in this field, as well
as of strong evidences that prove the better suitability of a
measure rather than another one, made us believe that such a
starting point for this research is sensible anyway.

*A. Data set*

The data set is built on 10 projects developed from 2003 to
2009 by the above cited company; it develops and maintains
a good number of software systems, mainly for local public
administration offices. We obviously focused our attention on
the applications written using web technologies, which are
now the most used by the company for developing its projects.
The vast majority of them are built with J2EE technology,
with a .NET exception. The analysed projects involve various applicative domains, but most of them are quite similar
information-centred systems, with a simple business logic
whose main task is performing computations and generating
reports on data inserted by the user, and retrieved from the
database.

For each project, we received the requirements documentation,
as well as the detailed Function Point estimate and some

pictures describing the layout of the web pages. These artefacts
constitute the basis for estimating the projects’ size expressed
in Web Objects. In that way, we ”simulate” an early project
phase, when only the requirements and some other kind of
documents have been produced; in fact, we used Function
Points estimates only at the end of the process for calculating
the total Web Objects (as they are part of the sum). Their


knowledge, as well as the knowledge of the actual effort,
should not bias the measurement.

*B. Web applications sizing*

Function Points [3] are the most widely used size metric
for ”traditional” applications; five function types are defined,
grouped in data (interface logical files, external interface files)
and transactions (external inputs, external outputs, external
inquiries). The counting of the number of function types, and
their weighting according to their complexity, returns a sum
called ”Unadjusted Function Points”. This is the measure that
the company computes for an early estimation of the projects,
neglecting a factor called VAF (Value Adjustment Factor) that
better calibrates the measure with more information about the

project, but at the price of a more complex procedure. A
further factor that the company considers for a more precise
weighting is the reuse level.
Web Objects [4] size measure can be considered as a significant contribute to web-based applications measurement field.
It adds four new elements, related to web development, to
the five FP functions (Figure 1). Then, complexity weighting
rules are applied, and the sum of these weights becomes the
functional size of the web application. The new elements
describe the multimedia content of the pages (Multimedia
files), the blocks of different nature that compose the pages
(Web Building Blocks), as well as embedded Scripts and Links
to external applications and databases.
In addition, we tried to define in a more univocal way the
several counting conventions specified in Reifer’s work, in
order to have a measure able to be as much consistent and

replicable as possible.

*C. Effort prediction*

The company does not extensively use any off-the-shelf
effort prediction model, like COCOMO, nor it developed
any model based on past projects. Instead, a productivity
coefficient, expressed in FP/man-day, is determined on the
basis of the technology adopted in that project; it varies from

1 (in some projects using standard J2EE technology) to 10
(when a project is developed using RAD tools). Since the
purpose of this research is the evaluation of size measures,
and not of cost models (which has been done in [18]), we
will apply this simple linear model also to the size calculated
using Web Objects, in order to compare in a consistent way
the two measures.

In practice, starting from the requirements of the ten examined
systems, we kept the original FP evaluation, and added to
it the specific evaluation of Web application characteristics,
according to Web Objects approach (Multimedia files, Web
Building Blocks, Embedded Scripts and Links to External
Applications and Databases). Clearly, the new estimate using
Web Objects is consistently higher than the orignal FP one,
because “points” were added, and never subtracted. To correct
for this systematic bias, we computed for each project the
percentage increase of Web Object estimate with respect to FP,
and averaged the ten values. We got a 58% average estimate
increase, so we rescaled Web Object estimates, dividing them
by 1.58. We kept instead the original productivity coefficient,
varying with the technology adopted in the project.

*D. Evaluation Method*

The estimation performances of the two measures in the
research context has been evaluated by calculating the MRE
(Magnitude of Relative Error) for each project. MRE is defined
as follows:

By performing *N* total observations, if *k* are the number of
observations with an MRE less than or equal to *l*, *Pred* ( *l* ) is
the percentage of projects with a MRE less than or equal to *l* .
As literature suggests [19], having a *Pred* (0 *.* 25) greater than
0.75 (more than 75% of the projects have an MRE less than
0.25) indicates an acceptable estimation, so we have a further
hint for evaluating the obtained results.

Results show a similar error for the estimation performed
using FP and Web Objects (rescaled), with a slight vantage
for FP. Note that WO are consistently better than, or equal
to FP in estimating Web applications using a RAD approach,
while they are typically worst for standard Java and .NET
applications. On the other hand, neither FP nor WO could
give us an acceptable estimation according to the criterion
enounced by [19], even though Function Points are much
closer to the 75% goal. One project was perfectly estimated
by FP, while another one highlighted the highest error for both
measures (respectively, 3.13 and 3.39); in that case the actual
effort was much lower than the estimated one, thanks to a
particularly skilled team. In general, both Function Points and
rescaled Web Objects counts resulted on average in an underestimation of the effort. FP estimation, on average, is more
precise that WO for the considered cases.
On the whole, these results are something unexpected, since

[18] obtained opposite results, from a similar context. The
main reason of these result can be attributed to the cost

model adopted by the company. This model, making use of
only Unadjusted Function Points and biult upon the company
expertise of many previous similar projects, has been used with
good results, despite its semplifications. On the other hand,
when another size measure, like Web Objects, is adopted, it is
not surprising that the same correction coefficients do not work
as well, despite the correction made to make them consistent
with the original FP method. While WO approach is not bad
with respect to the unadjusted FP used, in the considered
context it could be made better after a proper tuning of WO
points with respect to a new productivity coefficient depending
on the technology adopted. This of course needs quite a lot
of work, and enough time.

This study presents some limitations. The small size of the
data set, as well as the projects belonging to only one company,
can reduce external validity. Furthermore, the Web Objects
estimation were performed after the projects development,
although this problem was mitigated by the effort we spent
to limit bias ( *a priori* knowledge about the projects) in our

measurements.

V. C ONCLUSION AND FURTHER WORK

The most effective way of evaluating the validity of a
size measure is to assess its performance in estimating the
development effort required by a software project. This is
why we chose to apply Function Points and Web Objects
as input for the same (and already in use) cost model. To
perform the comparison, we obviously rescaled WO estimate
to match average FO estimates. The first results of this
research showed that, under these conditions, Web Objects
are slightly worse than Function Points in providing a precise
size estimate of a web application. However, these are only
partial results, and further experiments, using different cost
models, are required. Only the application of a more precise
model (based on regression, analogy, or specifically tailored
for web applications like WebMo) can tell us whether Web
Objects can perform equally well, or better, with respect to
Function Points. The facts that, in some cases such as RAD
development, WO performs better than FP, and that WO is
better correlated to system size measured in LOC for Java
J2EE applications, encourage to continue this research.
The next step of the research will be the development or
the choice of a cost model more suited to take as input a
size expressed in Web Objects. In that way, we will also be
able to compare the obtained results with those of the unique


similar study [18], that in fact developed an ex-novo cost
model based on Ordinary Least Squares Regression. Clearly,
a further validation of the results with a wider data set is also

required to improve the knowledge in this quite new research
field.
