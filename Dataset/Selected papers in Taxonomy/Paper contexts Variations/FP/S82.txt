1. Introduction


a b s t r a c t

Web development is moving towards model-driven processes whose goal is the development of Web applications at a higher level of abstraction based on models and model transformations. This brings new opportunities to the Web project manager to make early
estimates of the size and the effort required to produce Web applications based on their
conceptual models. In the last few years, several studies for size and effort estimation have
been performed. However, there are no studies regarding effort estimation in model-driven
Web development. In this paper, we present the validation of a model-based size measure
(OO-HFP) for Web effort estimation in the context of a model-driven Web development
method. The validation is performed by comparing the prediction accuracy that OO-HFP
provides with the accuracy provided by the standard function point analysis (FPA) method.
The results of the study (using industrial data gathered from 31 Web projects) show that
the effort estimates obtained for projects that are sized using OO-HFP are more accurate
than the effort estimates obtained using the standard FPA method. This suggests that by
following a model-driven development approach, the size measure obtained at the conceptual model of a Web application can be considered a suitable predictor of effort.

Web development is significantly different from traditional software development [44]. The traditional balance between
the project tradeoffs of time, cost, and quality has changed. The nature of Web development forces project managers to focus
primarily on the time variable in order to achieve the required short cycle times. Because of this imbalance, several problems
such as exceeding budgets and bad product quality have been reported in Web projects [16].
To address these issues, model-driven development approaches [43] seem to be very promising since Web development
can be viewed as a process of transforming models into successive ones until they can be executed in a target platform. Models can represent different abstraction levels of a Web application, starting from a platform-independent model to a platform-specific model, and finally to a code-level model. Over the last few years, a significant number of Web development
methods that follow a model-driven approach have been proposed (e.g., OOHDM [50], WebML [10], OO-H [22], W2000

[7]). Although most of these methods were developed in academic contexts, they have also been successfully applied in several industrial settings, including, e.g., ACER in Europe, the Middle East, Africa, and the Pan American regions, as well as
several banking institutions across Spain [40].

Adopting such methods, however, poses new challenges to the Web project manager, especially with respect to resource
estimation and project planning. A fundamental problem in this context is the size and effort estimation of the future Web
application based on its conceptual model. As suggested in [48], analysis objects (i.e., number of methods) are significantly
correlated with the size of a software product. Other approaches adapted standard functional size methods (e.g., COSMIC,
FPA) for sizing Web sites and applications [9,11,15]. However, the main limitation of these approaches are the following:
(i) they rely on design and/or implementation decisions; (ii) they are not automated [9,11,23,44], which limits their use
and adoption; and (iii) their conformance to standards relies heavily on the expertise of the measurer.
In order to deal with these issues, in [2], we proposed a measurement procedure for Web applications (object-oriented
hypermedia function points (OO-HFP)), which is compliant with the FPA method. A measurement procedure is defined as
a ‘‘set of operations, described specifically, used in the performance of particular measurements according to a given method
of measurement” [27]. The aim of OO-HFP is to size Web applications that are developed following a model-driven development approach using the object-oriented hypermedia (OO-H) method [22].
The main objective of this paper is to validate the OO-HFP for effort estimation in model-driven Web development by
comparing the prediction accuracy that it provides with the accuracy provided by the standard FPA method. The reason
for validating OO-HFP as a Web effort predictor is the need for an accurate effort predictor at the conceptual model level
of a Web application. Although some recent studies [17,49,37] have compared the accuracy of some Web size measures
as effort predictors, none of these measures have been validated in the context of a model-driven Web development process.
Thus, the main contribution of this work is to present an empirical study that validates the usefulness of a model-based size
measure for effort estimation in a specific model-driven development process. We believe that this paradigm gives the estimator a new and earlier chance for estimating size and effort in an accurate way, since modeling is in fact, programming at a
higher level of abstraction.
The reason for choosing the standard FPA is that it has been widely used in industry. The International Software Benchmarking Standards Group [24] provides a large database of industrial software and Web projects where this size measure
was employed. Since OO-HFP was proposed in order to facilitate the size estimation of Web applications when applying
the FPA method, we can verify whether or not OO-HFP is more accurate than FPA for estimating the size of Web applications.
These two measures are compared using data gathered from 31 Web projects developed by a Web development company.
We used both size measures as effort predictors in regression models.
This paper is organized as follows. In Section 2, we present an overview of related work on functional size measures for
Web applications with special emphasis on their use for Web effort estimation. In Section 3, we briefly introduce the reader
to the OO-H development process. In Section 4, we present an overview of the OO-HFP, which is followed in Section 5 by its
application to a case study. In Section 6, we show the validation of OO-H as a Web effort predictor. Finally, in Section 7, we
present conclusions and outline future work.

2. Related work

In the last few years, several Web measures to estimate the size of Web applications have been proposed. Some representative measures can be found in [38,39,32,15,6]. However, to date, there are no widely accepted Web measures in industry. Among these measures, we believe that a functional size measure could provide a good solution to the problem of sizing
Web applications since: (i) it is based on the functionality that will be provided to the users and not on the artifacts that are
produced when the Web application is delivered (e.g., number of Web pages, multimedia files); (ii) it can be measured in
early stages of the Web development process; (iii) it is based on standards that define the concept of size and the requirements for software sizing (e.g., [25,26]); and (iv) it is widely accepted and used in industry [24]. In the following, we review
existing functional size measures for Web applications as well as their use in Web effort estimation.

2.1. Functional size measures for Web applications

Most Web development methods break up the application’s design in three models: content, navigation and presentation.
The content model defines the application’s underlying structure and behavior. Content is commonly represented using UML
class diagrams or entity-relationship (ER) diagrams. The navigation model builds the nodes and links from content model’s
classes and relationships, and finally, the presentation model defines the abstract graphical interface from each node, thus
defining the appearance of the Web pages by means of abstract user interface elements.
The functionality represented in content models can be estimated by applying traditional size measures based on UML
diagrams (e.g., class points [14], Uemura et al. [52], object-oriented function points [5]). These measures were conceived
to estimate the size of object-oriented systems based on design documentation. Therefore, they can be used to estimate
the static and behavioral views of Web applications. For instance, the class point method is specifically focused on classes.
The method considered four different types of classes (e.g., classes that offer functionality for data storage and retrieval). The
complexity of each class is derived from the information about the local methods in the class, the interaction of the class with
the rest of the system, and, when available, the attributes [14]. However, these alternative size measures do not provide rules
for measuring the specific properties of Web applications commonly represented in navigation models and abstract user
interface models that are built upon content models.

This change in development practices makes the existing size measures for object-oriented systems not sufficient to
properly estimate the size of Web artifacts. Indeed, to cope with size estimation in Web development, several functional size
measurement procedures have been proposed. The first attempt to apply functional size measures to Web artifacts were the
guidelines proposed by IFPUG [23]. However, some practitioners [51] stated that these guidelines did not resolve many of the
counting issues faced when sizing Web applications. The practitioners provided interpretations for the IFPUG guidelines and
explained how FPA can be applied to Web artifacts. Rollo [47] also discussed some of the difficulties when measuring Web
applications with FPA, such as the difficulties encountered in identifying the system’s boundary and its logical files. Rollo

[47] argues that COSMIC is more flexible for counting Web applications. However, no evidence of this comparison was
provided.
Widely adopted FSM methods such as FPA and COSMIC do, in fact, have some relevant limitations. The first problem is
that the application of the rules requires human interpretation (e.g., it is not always clear how every element of the requirements should be classified and counted), consequently, an expert on applying the method is required (e.g., a certified function point specialist). The second problem is that none of the ISO-standardized FSM methods have been designed to take the
particular features of Web applications into account.
To address these limitations, 10 size measurement procedures for Web applications have been proposed in the last few
years. Of these procedures, four were proposed by practitioners (IFPUG guidelines for Web applications [23], Web-points

[11], Web objects [46], internet points [13]). The remaining six were proposed by researchers (OOmFPWeb [3,4], data
web points [41], Cândido and Sanches [9], Costagliola et al. [15], Fraternali et al. [20] and OO-HFP [2]). These measurement
approaches are summarized and listed chronologically in Table 1.
On the other hand, the size measurement procedures specifically proposed for Web applications (see Table 1) suffer from
some drawbacks. With the exception of Web-points [11], OOmFPWeb [4], and Fraternali et al. [20], they do not present a
well-defined set of mapping and measurement steps to apply functional size measurement to Web artifacts. Also, some
of the approaches (e.g., Web objects [46], Web-points [11], and internet points [13]) cannot be used in the early stages of
the Web development lifecycle since they rely on decisions regarding implementation languages and technologies. Furthermore, some approaches like Web objects [46], Web-points [11], and data Web-points [41] do not use an ISO-standardized
measurement method, but rather define new methods of sizing. One disadvantage of this is that the size measured with
these approaches is not comparable with the size standards that are currently accepted in practice.
All the proposals provide rules to size the static dimension of a Web application (i.e., the structural aspects commonly
represented in a data/object model). In [15], a preliminary mapping between Conallen’s UML extension and COSMIC is proposed. The procedure is based on counting data movements (to and from Web servers) and turns out to be suitable for capturing the specific aspects of dynamic Web applications. Some approaches also provide measurement rules for measuring
the navigation and/or presentation viewpoint. This is the case of Web objects that take into account the number of links
(xml, html and query language lines) as part of the size of a Web application.
However, only OOmFPWeb [4], Fraternali et al. [20], and OO-HFP [2] provide measurement rules to size all the dimensions of a Web application (i.e., data, behavior, navigation, and presentation). In fact, these proposals are somewhat similar.
OOmFPWeb [4] was proposed as a size measurement procedure for Web applications as a mapping between the OOWS conceptual modeling primitives [45] and the FPA concepts, whereas Fraternali et al. [20] and OO-HFP applied the same strategy
but using the WebML and OO-H conceptual modeling primitives, respectively. Of these methods, only WebML and OO-H
have been employed in industrial settings. This suggests that only the size measures defined for these methods can be properly validated in practice given the availability of industrial project data.

Apart from these problems, few evaluations of the validity of the proposed FSM alternatives for Web applications have
been documented in the literature. The data web points procedure [41] was tested using graduate and undergraduate students in computer science who were taking a regular course in software engineering. The course instructor and two industry
experts evaluated the quality of the size measures obtained with data web points [41] in a subjective manner. The size estimation was evaluated as: good, if it was very similar to the expert’s opinion; medium, if the estimation and the expert’s opinion were close; and poor, if they were not similar. The results for 22 Web projects were reported: 15 projects had good size
estimation; five projects had medium size estimation (two of these projects needed minor adjustments, and three of them
were incomplete); and two projects had poor size estimation (both were incomplete).
In [9], 20 Web applications were counted using the IFPUG FPA method, the NESMA FPA method (both the estimative and
indicative counting procedures), and the simplified method proposed in that work. This simplified method is based on
assigning a low complexity to all data and transactional functions. The results of the study found that the estimates using
the simplified method were close to the ones using the IFPUG FPA method (a difference of 4% was observed).
OOmFPWeb has been empirically validated using an evaluation framework for FSM methods [1,3]. The results of a family
of experiments with students showed that OOmFPWeb is efficient when compared to current industry practices. The results
also showed that the procedure produces reproducible size measurements and that it is perceived as easy to use and useful
by the study participants (who also expressed their intention to use this procedure in the future). However, OOmFPWeb
could not be validated for Web effort estimation given the lack of industrial Web projects obtained with the OOWS
methodology.
In [20], a comparison between manual and automatic counting using two different algorithms (pattern-based and duplicate pruning) for four different Web applications was reported. The results for the automatic counting using the duplicate
pruning algorithm differed from the manual counting by a maximum of 11%. This difference was detected in the duplicate
pruning algorithm due to its inability to identify some complex design patterns. However, Fraternali et al.’s procedure [7] has
not yet been validated for Web effort estimation purposes.

2.2. Web effort estimation

With regard to Web effort estimation, several studies have been reported in the literature [39,49,15,17,19]. A few of these
studies compared the effectiveness of some of the above-mentioned measures with other general-purpose measures. For instance, in the Ruhe et al. [49] study, the Web objects method was applied and validated for the first time. In this work, effort
estimation models based on Web objects and FPA were compared using ordinary least square regression. The authors of that
study found that the performance of Web objects was superior to the FPA standard method.
In another study, Costagliola et al. [15] reported on an empirical analysis about the effectiveness of COSMIC to predict
Web application development effort. The study used 44 Web applications developed by students. The results of the study
suggest that the counting of data movements can be useful for estimating the development effort of dynamic Web
applications.
Similarly, Di Martino et al. [17] compared the effectiveness of four Web size measures (i.e., Tukutuku measures, Web objects, length measures, and functional measures) for predicting Web application development effort. The results of the study
(using data from 15 Web projects developed by a software company) showed that all the measures provided good effort predictions. In another study, Ferrucci et al. [19] compared COSMIC and Web Objects in order to verify their effectiveness as
indicators of development effort. The results of a study including a dataset with 15 Web applications reveal that both
Web objects and COSMIC are good indicators of the development effort.
However, despite their promising results, the above studies are inconclusive with regard to identifying the best effort predictor in Web development. In other words, there is no single effort model that can be applied in all situations or company
circumstances. Furthermore, there is a lack of studies on effort estimation for model-driven Web development. To the best of
our knowledge, there is only one study performed in this context [2]. In this study, we presented a preliminary evaluation of
OO-HFP as a Web effort predictor by comparing the prediction accuracy that it provides with the accuracy provided by another set of validated Web size measures (the Tukutuku measures [37]). The results of that study, based on the analysis of 12
industrial Web projects, showed that the effort estimates obtained for projects that were sized using OO-HFP were similar to
those obtained using the Tukutuku measures, suggesting that OO-HFP is a suitable Web effort predictor.
However, more experimentation is needed to establish a body of knowledge in the field. These experimentations should
be performed to confirm the previous empirical results. Here, we apply and validate the OO-HFP approach for effort estimation purposes in the context of a Spanish Web development company that employs a model-driven Web development method (OO-H).

3. The object-oriented hypermedia method

The object-oriented hypermedia (OO-H) method [22] provides the semantics and notation for developing Web applications. The main feature of this method is its model-driven and transformation-based approach. Following, we introduce the
OO-H development process and its metamodel.

3.1. The OO-H development process

According to the model-driven architecture approach defined by the object management group [43], the OO-H development process (see Fig. 1a) starts by defining a requirements model (computation-independent model, CIM). Skeletons of platform-independent models (PIMs) can be obtained from these requirements and then manually completed to represent the
main dimensions of a Web application: content (UML class diagram); navigation (navigation access diagram, NAD); and presentation (abstract presentation diagram, APD).
OO-H associates a different NAD to each type of user. The NAD captures the navigation paths and services that the user
can activate when working with the Web interface. From the UML class diagram and the NADs, a Web interface can be generated without further work because OO-H method provides a set of default values for the main presentation features. This
allows the designer to shorten the time needed to develop Web applications (to get more information about the NAD and the
OO-H method, interested readers are referred to [22]).
However, once an agreement between the designer and the stakeholder has been reached, the designer will most probably need to modify this default Web interface in order to improve both its appearance and usability features. In order to do
this, OO-H method defines another diagram: the abstract presentation diagram (APD). This diagram is obtained by merging
the class and navigation diagrams. Finally, platform-specific models (PSMs) are automatically obtained from the APD, which
can be manually modified in order to adjust to the user interface requirements. Source code can then be generated from the
APD to obtain the Web application. The aim of this model-driven development process is to automate the definition and
transformation of models based on a set of predefined transformation rules that are specific for each target platform. The
VisualWADE tool [53] automates the entire OO-H development process.

3.2. The OO-H metamodel

This section explains the main concepts of the OO-H metamodel [22] that are used to represent navigational properties
(see Fig. 2). The navigation model of a Web application is defined by a set of navigational maps. Each navigational map structures the navigational view of the system for a specific type of user and has a unique entry point that indicates the starting
point of the navigation process. A navigational map is made up of a set of navigational elements that can be specialized as
navigational nodes and/or navigational links. A navigational node represents a view over the class diagram. A navigational
node can be a navigational target, a navigational class, or a collection.

A navigational target groups some elements of the model (i.e., navigational classes, navigational links, collections) that collaborate in the coverage of a user navigational requirement. A navigational class represents a view over a set of attributes
(navigational_attribute) and operations (navigational_operation) of a class from the class diagram. A collection is a hierarchical
structure that groups a set of navigational links. A navigational link defines valid node attainability inside the navigational
map. Navigational links define the navigation paths that the user can follow through the user interface. Each navigational
link has an origin (org) node and its corresponding destination (dst) node. There are two types of links: internal links to define
the navigation path inside a navigational map, and service links to show the services available to the user associated to that
navigational map.
In addition, OO-H allows the use of the object constraint language (OCL) to define constraints over the class diagram and
the NAD. In the NAD, such constraints are expressed by means of filters that are defined upon links (internal or service).

4. OO-HFP overview

The OO-HFP [2] is a FSM procedure that conforms the standard IFPUG FPA method [25]. It takes into account the domain
(UML class model) and the navigation (NAD) views as input to calculate the size. These views contain all the elements that
contribute to the functional size of a Web application.
Given the conformity of this procedure to FPA, these elements are, in essence, the same as the ones described in the FPA
metamodel; they basically distinguish five function types: internal logical files (ILF), external interface files (EIF), external
inputs (EI), external outputs (EO), and external inquiries (EQ). These elements are identified based on a collection of mapping
rules (see Section 4.1). OO-HFP also provides another set of rules, the numerical assignment rules (see Section 4.2), which are
used to identify the elements that add to the complexity of the identified function types. Finally, the IFPUG tables are used to
rate the complexity of the functions, to assign weights to the elements of the model, and to aggregate the assigned values
into an overall functional size value for the Web application. The calculation of the OO-HFP size measure was automated in
VisualWADE [53], which is a tool that supports the OO-H development process. This avoids the ambiguity of interpreting the
FPA counting rules and the need for special training to count function points in an accurate and repeatable way.

4.1. The mapping rules

The mapping rules help to identify the elements in an OO-H conceptual model that contribute to the size of the Web
application. These rules (see Table 2) are defined as a mapping between the concepts of the FPA metamodel onto the concepts in the UML/OO-H metamodel [42,22]. The use of metamodels in software measurement has shown to be effective [21].
First, the rules to establish the counting scope and the boundary of the Web application are applied (Rules 1–4). The
counting scope defines the functionality which will be included in a particular function point count. It defines a (sub)set
of the software being sized. To properly define the counting scope, the user view should be previously established. In fact,
the user view defines how the user sees the application. This is an important activity before measurement because in a Web
environment this view can vary. For instance, a Web portal offers a variety of Internet services from a single location. It can
be viewed as different applications in the same location. Thus, the user can view the portal as a single application or as separate applications.

Therefore, how can a correct counting scope be defined? This is one of the problems to be deal with when designing functional size methods for Web applications. In our approach, this problem is solved since we measure conceptual models that
were previously defined from the user point of view. Using the OO-H conceptual model primitives, one or more applications
can be designed in accordance with the business goal and the requirements of the Web application.
In OO-H, a navigation access diagram (NAD) is built to represent the information and the functionality that a type of user
can access as well as the valid paths that she/he can go through the Web application. Therefore, a navigation access diagram
represents the view that a particular type of user has of the Web application. As a result, the counting scope includes the
UML class diagram as well as the navigation access diagrams that are built for each type of user.
The boundary indicates the border between the software being measured and the user. In OO-H, each class of the UML
class diagram with the ‘‘actor” stereotype is considered to be a user type of the Web application whereas each class with
the ‘‘legacy” stereotype is considered to be an external application. Therefore, the boundary corresponds to an imaginary line
traced in the UML class diagram. The ‘‘actor” and ‘‘legacy” classes are considered to be outside the boundary, whereas the
other classes are considered to be inside the boundary.
Once the counting scope and the boundary are established, the data (ILF and EIF) and transactional (EI, EO, and EQ) functions are identified. In FPA, an ILF is defined as a user identifiable group of logically related data or control information maintained within the boundary of the system. In OO-H, this corresponds to a class that encapsulates a set of data items
(attributes) representing the state of the objects in each class. Aggregation and composition hierarchies can also be considered an ILF since they represent a unique group of logically related data (Rules 5–7).
An EIF is a user identifiable group of logically related data or control information referenced by the system, but maintained
within the boundary of another system. In OO-H, this is a special type of class with the ‘‘legacy” stereotype which allows the
connection with existing external applications (Rule 8).
An EI is an elementary process that manipulates data or control information that comes from outside the system boundary. Its
goal is to maintain one or more ILFs and/or to alter the behavior of the system. In OO-H, this corresponds to a method of a
class, since it always changes the state of the objects of the class. However, a method must have public visibility and must be
identified only once (in the declared class), even if it is inherited (Rule 9). Methods in a class with public visibility can be seen
and used from outside the system boundary. The parameters of these methods specify the data that cross the system
boundary.
An EQ is an elementary process that sends data or control information outside the system boundary. Its goal is to present information to a user through the retrieval of data. In OO-H, this corresponds to a navigational target defined in the NAD (and each
abstract page derived from it) since it may describe different kinds of user interactions with the Web application, where data
generated within the system is presented to the user.
A navigational target may derive one or several abstract pages depending on the value of the property ‘effect’ of a navigational link. The value source (a light arrow) indicates that the information specified in the navigational target will be presented in the current abstract page; the value target (a dark arrow) indicates that the information will be presented in a
different abstract page (Rule 10). For an example of these types of navigational links see Fig. 4.
Finally, an EO has the same definition as an EQ, but its processing logic must contain at least one mathematical formula or
calculation, or it has to create derived data. In OO-H, this also corresponds to the concept of navigational target. However, at
least a class in the navigational target must include one derived attribute (Rule 11). Derived attributes are attributes whose
values are generated from other attributes using calculations, algorithms or procedures. Thus, we assume that the difference
between EQ and EO is that an EQ does not contain derived or calculated data.

4.2. The numerical assignment rules

The numerical assignment rules help to produce a quantitative value that represents the Web application functional size.
This is accomplished by applying two sets of rules that are introduced in the following subsections.

4.2.1. The OO-HFP measurement rules

These rules are used to identify the elements that add to the complexity of the identified functions (e.g., DETs, RETs, FTRs).
According to the FPA metamodel, the complexity of a class (ILF) or ‘‘legacy” class (EIF) is determined by counting the number
of data element types (DET) and the number of record element types (RET). The measurement rules proposed to identify the
DETs and RETs of a class are shown in Appendix A, Table A-1. A DET is identified for each single attribute of a class, and a RET
is identified for the class itself. In addition, we consider both aggregations/compositions and generalization/specialization
relationships as contributing to the complexity of a class. This is because the FPA method suggests counting one DET for each
piece of data that indicates a relationship with another ILF or EIF.
Associations, aggregations, and compositions are measured according to the multiplicity property of the relationship. This
property specifies the lower/upper number of objects that can be associated to a single object of the class. If the number of
instances in the target relationship is one, one DET is counted. Otherwise, if the number of instances in the target relationship is greater than one, one RET is identified. Specialization relationships are measured in the subclasses by counting one
DET for each super class. Finally, the complexity of aggregation, composition, or inheritance hierarchies is measured by
counting one RET for each class in the hierarchy. The complexity of a method (EI) or navigational target (EO and EQ) is determined by counting the number of data element types (DET) and file types referenced (FTR).
Some representative measurement rules for the complexity of a method of a class are shown in Table A-2. Each single
argument of a method is counted as one DET, and each object-valued argument is counted as one FTR. Two DETs are counted
in order to be compliant to FPA: one DET for the capability of the application to send a response message outside the boundary (error, confirmation, control) and another DET for the ability to specify an action to be taken. One FTR is also counted for
the class where the method is declared. Additional FTRs are also counted for each one of the distinct classes that appear in
the OCL formula (e.g., precondition, postcondition) of a method of a class.
Finally, some representative measurement rules for the complexity of a navigational target are shown in Table A-3. A navigational target groups elements (navigational classes, navigational links, and collections) to fulfill a user navigational
requirement. One DET is counted for each element in a collection that implies navigating to another collection.
Additional DETs are counted for each visible attribute or method of a navigational class. One DET is counted for each link
between navigational classes. This is because FPA suggests counting one DET for each piece of data that indicates a relationship with another group of data. Two other DETs are counted to be compliant to FPA: one DET for the capability of the application to send a response message and another for the ability to specify an action to be taken.
One FTR is counted for each navigational class of the navigational target. Additional FTRs are counted for the classes that
appear in the OCL formula of a navigational class or navigational link (e.g., precondition, filter). There are also other measurement rules for navigational patterns (e.g., index, guided tour) that can be specified in a navigational link. The complete set of
rules is described in the OO-H measurement guide [44].

4.2.2. The standard FPA counting rules
These rules are used to rate the complexity of the data and transactional functions, to assign weights to these functions,
and finally, to aggregate the assigned values into an overall functional size value for the Web application. The ISO/IEC 20926
standard [25] provides several tables to determine the complexity levels of a function type. As an example, Table 3 shows the

Then, a value in function points (FP) is assigned to each function depending on its type and complexity level. FPA also
provides several tables to assign FP values for the identified functions. The proposed values for each function type are as

� ILF: low = 7 FP, average = 10 FP, and high = 15 FP.
� EIF: low = 5 FP, average = 7 FP, and high = 10 FP.

5. Applying OO-HFP

Fig. 1 contains a representation of the procedure that is used to apply OO-HFP showing the relationships among OO-H,
OO-HFP, and FPA. The Web size measurement process is represented according to the generalized representation of functional size measurement proposed by Fetcke et al. [18] (see Fig. 1b). This representation was applied to represent several
FSM methods such as IFPUG FPA, Mark II FPA, and full function points [18]. According to this model, functional size measurement requires two steps of abstraction: identification and measurement.
OO-HFP conforms to this generalized representation of functional size measurement. During the identification step (using
as input the OO-H conceptual model), an abstraction of the modeled Web application is made using the OO-HFP mapping
rules. This abstraction contains only those elements of the application that are relevant to functional size measurement,
according to the FPA metamodel. Next, during the measurement step, the OO-HFP measurement rules are used as part of
the process of assigning numbers to the elements of the abstract model.
For each step of the OO-HFP measurement procedure, we also show the equivalent step in the process model proposed by
Jacquet and Abran [28] (see Fig. 1b). According to this model, three steps are suggested for an application of the measurement method: software documentation gathering, construction of the software model, and application of the numerical
assignment rules. In the remainder of this section, we follow this process model to illustrate the use of the OO-HFP measurement procedure in a case study.

6. The validation of OO-HFP for effort estimation

The validation of OO-HFP for Web effort estimation is done by comparing the prediction accuracy that it provides with the
accuracy provided by the traditional FPA measure. The results presented in this section were obtained using the statistical
software SPSS 15.0.1 for Windows.

6.1. Validation method

The validation of the prediction accuracy of the size measures is performed as follows [35,31,33]:

� Each of the two size measures is used as an independent variable to build an effort estimation model using a simple Linear
Regression. Linear Regression [34] is a statistical technique that is used to build a prediction model (Equation). The equation represents the relationship between independent and dependent variables. We selected this technique because it
provides accurate results when estimating Web projects (e.g., [49,37,15,17,35,19]). In our study, we predict the effort
needed to develop a Web project based only on the knowledge of its size. Therefore, we employed OO-HFP and FPA as
independent variables and ActualEff as a dependent variable.
� The stability of each effort estimation model is verified by following the procedure proposed by Mendes [35], which consists of the following steps:
� The use of a residual plot showing residual vs. fitted values. The goal is to determine whether the residuals are random
or normally distributed.
� The use of Cook’s distance values for all projects to identify influential data points. With this procedure, any projects
that have distances greater than 3 � (4/n) (where n represents the total number of projects) are removed from the data
analysis. Those projects that have distances greater than 4/n but smaller than (3 � (4/n)) are removed in order to test
the model stability, which is done by observing the effect of their removal on the model. If the model coefficients
remain stable and the R [2] improves, the influential projects are retained in the data analysis.
� The effort predictions of the final regression models obtained are measured using the mean magnitude of relative error
(MMRE), median MRE (MdMRE), and Pred (0.25). These are commonly used measures for the evaluation of estimation
models [12]. The magnitude of relative error (MRE) is calculated for each project whose effort is predicted:

applications.

6.2.4. Model validation

The accuracy of the effort estimates was assessed using a leave-one-out cross-validation procedure. Cross-validation is the
splitting of a dataset into training sets, which are used to build the effort estimation models, and validation sets, which are
used to validate the models. In a leave-one-out cross validation, the original dataset is divided into n different subsets (n is
the size of the dataset) of training and validation sets, where each validation set has only one project.
The cross-validation we employed was, therefore, a 31-fold cross-validation. This means that 31 training and validation
sets were used to obtain accuracy statistics. Each time, we used 30 projects in the training set and 1 project in the validation
set. The prediction accuracy of each model was measured using the Mean MRE (MMRE), Median MRE (MdMRE), and Pred
(25) statistics. We also compared each model with predictions obtained using the mean and median efforts of the dataset
in order to have a benchmark. Finally, as suggested by Maxwell [34], the statistical significance of the absolute residuals
for OO-HFP and FPA was checked using the Wilcoxon Signed Ranks test, with a = 0.05.

6.2.4.3. Threats to validity. In terms of the internal validity of our results, the FPA size measure was obtained manually by a
certified function point specialist using the IFPUG counting manual. The OO-HFP measure was well-documented with a measurement procedure, and the OO-HFP size measures were obtained automatically using the VisualWADE [53] plug-in that
implements this measurement procedure.
One factor that could affect the results of our study is the size of the dataset employed (31 Web projects). However, we
believe that this does not affect the validity of our results since we obtained significant statistical results. In addition, other
studies for effort estimation conducted in the field of Web engineering also obtained significant results with smaller datasets
of 12 and 15 Web projects [49,17,19].
In terms of the external validity, we employed a dataset that does not represent a random sample of Web projects. Consequently, our results can only be generalized to Web development companies that follow a model-driven Web development
process that is based on the OO-H method. Nevertheless, it can be considered a representative method of the whole set of

The fact that the full sample comes from a single company somewhat decreases the validity of our results. Although some

7. Conclusions and further work

This paper has presented the validation of a functional size measure (OO-HFP) for effort estimation in a model-driven Web
development process. This measure is compliant to FPA, which is a widely used FSM method in industry. The purpose of the
validation was to compare the prediction accuracy that OO-HFP provides with the accuracy provided by the standard FPA
method.

The results showed (at least for the dataset employed) that the effort estimates obtained for projects that are sized using
OO-HFP is more accurate than the effort estimates obtained using the standard FPA method. The results also showed that
OO-HFP provides better predictions than the ones based on the median or mean of the dataset. This indicates that by following a model-driven development approach according to the OO-H development process, the size measure obtained at the
conceptual model of a Web application can be considered a suitable effort predictor. This is in line with our previous results

[2], which were obtained by comparing OO-HFP with a general-purpose Web size measure (i.e., the Tukutuku measures

[37]). Thus, our results provide more evidence about the usefulness of a model-based Web size measure for effort estimation
purposes. Since models are considered to be the most important artifacts in model-driven development, instead of predicting
the amount of effort needed to develop a Web application based on the source code, we predicted it based on the corresponding conceptual models.
As future work, we plan to conduct further investigations using larger datasets of Web projects collected from culturally
and organizationally different Web companies to ensure the external validity. We also plan to replicate this study using
other effort estimation techniques (e.g., case-based reasoning) and other size measures (e.g., Web objects). Since the OOHFP was tailored to fit the OO-H method, which was defined following the MDA standards for model-driven development
(e.g., MOF, UML), many of the modeling primitives in the OO-H metamodel can also be found in other Web development
methods [7,10,50]. Therefore, we plan to investigate to what extent OO-HFP can be adapted for use in other model-driven
Web development approaches.
