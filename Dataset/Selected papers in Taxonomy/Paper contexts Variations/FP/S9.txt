**Abstract**

Software practitioners recognise the importance of realistic estimates of effort to the successful management of software
projects, the Web being no exception. Estimates are necessary throughout the whole development life cycle. They are
fundamental when bidding for a contract or when determining a project’s feasibility in terms of cost-benefit analysis. In
addition, they allow project managers and development organisations to manage resources effectively.
Size, which can be described in terms of length, functionality and complexity, is often a major determinant of effort. Most
effort prediction models to date concentrate on functional measures of size, although length and complexity are also
essential aspects of size.
The first half of this paper describes a case study evaluation in which size metrics characterising length, complexity and
functionality were obtained and used to generate effort prediction models for Web authoring and design. The second half
describes the comparison of those size metrics as effort predictors by generating corresponding prediction models and
comparing their accuracy using boxplots of the residuals. Results suggest that in general all categories presented a similar
prediction accuracy.

**1. Introduction**

In any scientific field, measurement generates quantitative descriptions of processes and products, making it possible to
understand behaviour and result. This understanding allows for the selection of better techniques and tools to control and
improve processes, products and resources [1].
As engineering involves the analysis of measurements [1], hypermedia engineering and in particular, Web engineering,
will only be considered a true engineering discipline when a solid foundation of measurement-based theories exist.
Software practitioners recognise the importance of realistic estimates of effort to the successful management of software
projects, the Web being no exception. Estimates are necessary throughout the whole development life cycle. They are
fundamental when bidding for a contract or when determining if a project’s feasibility in terms of cost-benefit analysis. In
addition, they allow project managers and development organisations to manage resources effectively.
Prediction is a necessary part of an effective process, be it authoring, design, testing, or Web development as a whole.

Size, which can be described in terms of length, functionality and complexity, is often a major determinant of effort. Most
effort prediction models to date concentrate on functional measures of size, although length and complexity are also
essential aspects of size.

This paper focuses on effort prediction for design and authoring processes. We adopt the classification proposed by Lowe
and Hall [2] where authoring encompasses the management of activities for the content and structure of the application
and its presentation. Design covers the methods used for generating the structure and functionality of the application, not
including aspects such as hypermedia application requirements elicitation, feasibility consideration and applications
maintenance. In addition, our design phase also incorporates the conceptual design of the application, presented by map
diagrams showing documents and links.

The data used to generate our prediction models was gathered through a quantitative case study evaluation where a set of
suggested size metrics for effort prediction were measured. A number of complexity metrics from literature in Software
Engineering [3] and Multimedia [4,5] were reused. In addition, several length metrics were adapted from [6]. The
Prediction models proposed were generated by statistical techniques, specifically Linear and Stepwise Regression.
Section 2 presents related work in development effort prediction for Web applications. Section 3 describes the case study
evaluation and Section 4 presents and compares the prediction models using boxplots. Finally, we give our conclusions
and comments on future work in Section 5.

**2. Related Work in Web Engineering**

To date there are only a few examples of effort prediction models for Web development in the literature. Most work in the
field of Web engineering concentrates on the proposal of methods, methodologies and tools as a basis for process
improvement and higher product quality [7,8,9,10].

Morisio et al. [11] describes an experiment involving the development of Web-based applications using an object-oriented
framework in which functional and code size metrics of the software produced were collected. The metrics were Lines of
code, classical Function Points and Object-Oriented Function points. Their data analysis revealed that learning effects
affected mainly the cost of writing new code. They applied an existing effort prediction model based on different reuse
types and concluded that classical function points were not appropriate for productivity estimations in the type of
environment they have used. Although their conclusions were quite interesting, the parameters of the models used only
apply to Web applications developed using an object-oriented framework.

Rollo [12] explores the issues of counting a Web site in both IFPUG [13] and the MKII [14] methods. He discusses
difficulties and contradictions between the rules and some approaches to Web site sizing. In addition, the same Web site is
counted using the sizing method COSMIC-FFP [1] [15]. His conclusions were that COSMIC proved to be the most flexible
approach for counting the functional size of Web sites and can be applied to any Web site.

Mendes et al. [16] describe a case study evaluation involving the development of Web sites structured according to the
Cognitive Flexibility [17] principles in which length and complexity metrics were collected. The metrics obtained were
hyperdocument size, connectivity, compactness, stratum and reused-documents. Several prediction models are generated
(linear and stepwise regression and analogy) and compared using the Mean Magnitude of Relative Error measure. Results
show that the best prediction were obtained using analogy. Their results were an initial step towards the proposal of Web
development effort prediction models. However, some of these metrics were subjective, which may have influenced the
validity of their results.

The three works briefly presented in this Section, although cannot necessarily scale to other environments, show
methodologies which may be applicable in other Web development environments.

**3.4 Measuring Size metrics**

Complexity and Length metrics were collected using a questionnaire and three spreadsheets which subjects were required
to complete. All complexity and length metrics, except reuse, were re-measured by using the applications developed.

Functionality was measured using the COSMIC-FFP [15] measurement method. We counted the number of Cosmic
Functional Size Units (CFSUs) in the Web application. The COSMIC-FFP method is a standardised measure of software.
It is applicable to domains such as application software (banking, insurance, accounting etc), real-time software, hybrids
of both. We have adapted the method to Web applications, in a way similar to [12].

The COSMIC-FFP method involves applying a set of rules and procedures to a given piece of software perceived from the
perspective of its Functional User Requirements (FURs) [7] . The result is a numerical "value of quantity" [8] representing the
functional size of the software. The model consists of two parts: the context model and the software model.

The context model helps establish what is considered to be part of the software and what is part of the software's operating
environment. It is necessary to identify the boundaries of the software and the functional flow of data attributes. The flow
of data attributes has two directions: front-end and back-end. In the front-end direction, two types of movement
(ENTRIES and EXITS) allow the exchange of data attributes with the users. In the back-end direction, two other types of
movement (READS and WRITES) allow the exchange of data attributes with the storage hardware (See Figure 2).

The counting rules used corresponding to each entry, exit, read and write are as follows:
! Each "HREF" tag counted as 1 entry + 1 read + 1 exit. By pressing a link, the user sends an entry to the application,
which reads the data from the Web server and shows the contents to the user. We assumed the Web server to be the

Data Storage Hardware. Although several subjects used in-line images [19], they were not considered as a separate
data group since every "IMG SRC" tag will be downloaded automatically with the Web page. The subjects who
participated in this case study did not use external resources (images, sound and video), which use the "HREF" tag
and have to be downloaded from the Web server upon request.
! In relation to the types of programs used, subjects used either Java applets or Javascript. For each Java applet we
counted one entry + one exit (one entry to activate the applet and one exit showing the applet). As applets are also
downloaded from the server at the same time as the Web page, we did not consider a "read" flow of data.
! For each Javascript file we only considered the "HREF" tag ("entry" flow of data) as it was the only source of
interaction between user and application.

We did not have a Functional Users Requirements documents for the Web applications, so we counted the number of
CFSUs for each application using the implementation provided by the subjects only. Some links ("HREF" tags) existed to
give readers more information about the contents of the application; others to give better navigation guidance (Site map,
back to main page, previous page, next page etc). We understand that both types of links are equally important in
achieving the purpose of conveying information in a meaningful way and a reasonable level of functionality.

! Design and authoring effort are difficult concepts to measure. However, it is argued that the data collected and
the response variable derived are intuitively reasonable measures.
! There is no standard on what constitutes the design and authoring processes of a Web application. Therefore, the

! The metrics collected, apart from effort, experience and structure, were all objective and precisely quantifiable.

**4. Estimation Process**

**4.1 Introduction**

For the collected data, linear regression and stepwise regression were used to generate effort prediction models for Web
design and authoring.

The purpose of linear and stepwise multiple regression models are to learn more about the relationship between several
predictor variables and a response variable. Multiple regression and linear regression allow the researcher to ask (and
hopefully answer) the general question "what is the best predictor of ..." [26]. Regression modelling is one of the most
widely used statistical modelling techniques for fitting a quantitative response variable *y* as a function of one or more
predictor variables *x* *1* *, x* *2* *,...,x* *p* . Regression models are widely used as they often provide an excellent fit to a response
variable when the true functional relationship between the response and the predictors is unknown [27]. Both linear
regression and stepwise regression were calculated using SPSS 10.0.5.

**4.3 Comparing the Prediction Models**

We used boxplots (see Figure 5) of the residuals to compare the two prediction models generated, for each one of the Size
categories, as suggested by Pickard et al. [28] and Kitchenham et al. [29]. It gives a good indication of the distribution of
the residuals and can help explain the behaviour of the summary statistics.

**Figure 5 - Main components of a boxplot [30]**

The box describes central tendencies of a distribution. The horizontal line in the middle of the box locates the median. The
edges of each box mark the 25th and 75th percentiles. The height of the box is the interquartile range, containing 50% of
observations. The vertical lines up or down from the edges (whiskers) contain observations which are less than 1.5 times
interquartile range. The boxplot also marks outliers. If the observation is more than 3 times the length of the box below
the lower whisker or above the upper whisker, it is called an extreme value. If the values are between 1.5 and 3 times the
height of the box, the observation is called outlier [30].

When upper and lower tails are approximately equal and the median is in the centre of the box the distribution is
symmetric. If the distribution is not symmetric the relative lengths of the tails and the position of the median in the box
indicate the nature of the skewness. The lenght of the box relative to the length of the tails gives an indication of the shape
of the distribution. A boxplot with a small box and long tails represents a very peaked distribution whereas a boxplot with
a long box represents a flatter distribution [29].

**5. Conclusions and Future Work**

The first half of this paper presented a case study evaluation in which we measured attributes of Web applications
corresponding to three size categories, namely Length, Complexity and Functionality. Although some of the suggested
metrics can be used in the prediction of effort for dynamically generated Web applications (Program count, Total Code
Length, Reused program count, Total reused code length, Size CFSU ), the metrics were mostly related to static applications.
The second half of the paper suggested effort prediction models generated for each one of the size categories, using two
statistical techniques, Linear regression and Stepwise Multiple Regression.

For each one of the prediction models generated, the estimated effort was calculated and subtracted from the actual effort
in order to obtain the residuals. Boxplots of the residuals were used to compare the predictions from the various prediction
models generated. None of the models generated presented reasonable accurate estimates of effort. In addition, there were
no statistical significance of the absolute residuals, which means that all samples were obtained from the same population.

At least for the Web applications used in our dataset it seems that measuring the functionality of those applications was
very much related to the number of links that the application has. As we counted the number of Cosmic Functional Size

12

Units using the final implementation, rather than the Functional User Requirement model, further investigation is clearly
necessary to determine whether or not functionality is highly correlated to connectivity.

Our future work includes using the COSMIC-FFP model to measure the functionality of several types of static and
dynamic Web sites and to use real datasets from industrial practice.

To conclude, there is an urgent need for adequate Web development effort prediction at an early stage in the development.
As the use of the Web as a resource delivery environment increases, effort estimation can contribute significantly to the
reduction of costs and time involved in developing Web applications.
